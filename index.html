문제 정의

매 배포마다 핵심 기능(로그인, 검색, 메뉴 접속 등) 수동 확인 반복. 테스트 실패 시 버그 일감 등록까지 별도 공수 필요 — 실제 대응보다 보고 과정에 더 많은 시간 소요.

02

해결 방법

  • E2E 자동화 — 로그인, 로그아웃, 검색, 메뉴 접속, 백오피스 로그인 등 핵심 시나리오를 Playwright 기반으로 자동화
  • 자동 일감 생성 — 테스트 실패 시 사내 일감 관리 도구에 버그 일감 자동 생성 (제목, 재현 경로, 스크린샷 포함)

  • 브라우저 UI — React / Node.js / TypeScript 풀스택으로 구현해 브라우저에서 테스트 실행 및 결과 확인 가능
03

성과

  • 반복 운영 체크 공수 80% 이상 절감
  • 테스트 실패부터 버그 일감 등록까지 흐름 자동화
  • Dev / Stage / Live 3개 환경 동시 검증 체계 구축

  • 플랫폼 기획 — 구매·내부 제작 자산을 통합 관리하는 사용자 화면과 백오피스를 기획
  • 프로젝트 관리 — 애자일 스크럼 기반으로 로드맵에 따라 마일스톤별 스펙을 선별하고 우선순위를 협의
  • 온보딩 — 온보딩 자료와 운영 가이드를 제작해 사용자 진입장벽을 낮춤

Insight

공유 취지와 현실 사이에서 기준 세우기

전사 자산 공유로 비효율을 줄이는 것이 목표였지만, 모든 조직이 공유를 반긴 것은 아니었습니다. 자산을 적극 공유하려는 조직도 있었고, 성과와 직결돼 공개를 꺼리는 조직도 있었습니다.

그래서 '무조건 통합·공개'가 아니라, 프로젝트별 데이터 접근 권한 제어를 먼저 설계했습니다. 공유가 민감한 조직의 자산은 비공개로 두되, 공유된 에셋은 누구나 볼 수 있게 하고 필요할 때 선택적으로 공유할 수 있는 구조를 만들었습니다.

결국 공유 효율과 자산 보호는 충돌하는 가치였고, 이해관계자의 우려를 정책과 운영 구조로 먼저 풀어내는 일이 서비스 기획자의 핵심 역할이라는 점을 배웠습니다.
